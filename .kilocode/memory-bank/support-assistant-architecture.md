# Архитектура мини-сервиса поддержки пользователей

## Обзор
Мини-сервис для поддержки пользователей, интегрированный в существующий llm-service (FastAPI + Ollama + RAG).

## Основные компоненты

### 1. Модуль CRM/User Data (`app/crm.py`)
- Работа с данными пользователей и тикетов
- Поддержка JSON файлов и потенциальная интеграция с CRM через MCP
- Структуры данных:
  - User: id, name, email, subscription_plan, created_at
  - Ticket: id, user_id, title, description, status, priority, created_at, updated_at
  - SupportHistory: история обращений пользователя

### 2. Модуль Support RAG (`app/support_rag.py`)
- Расширение существующего RAG для поддержки
- Индексация FAQ и документации продукта
- Интеграция контекста пользователя в поисковые запросы
- Поддержка поиска по истории тикетов

### 3. API эндпоинты (`app/support_api.py`)
- `/api/support/chat` - основной эндпоинт для ассистента поддержки
- `/api/support/users` - управление пользователями
- `/api/support/tickets` - управление тикетами
- `/api/support/faq` - управление FAQ

### 4. Интеграция с существующей системой
- Использование существующего OllamaClient
- Расширение существующих моделей Pydantic
- Интеграция с текущей системой аутентификации

## Поток данных
1. Пользователь задаёт вопрос через API
2. Система определяет контекст пользователя (если предоставлен user_id или ticket_id)
3. RAG ищет релевантную информацию в:
   - FAQ продукта
   - Документации
   - Истории тикетов пользователя
4. Контекст пользователя включается в промпт
5. Ollama генерирует ответ с учётом контекста
6. Ответ возвращается с источниками

## Конфигурация
- `SUPPORT_DB_PATH` - путь к БД поддержки
- `FAQ_PATH` - путь к FAQ файлам
- `CRM_TYPE` - тип CRM (json, mcp, external)

## Пример использования
```python
# Запрос с контекстом пользователя
{
  "messages": [{"role": "user", "content": "Почему не работает авторизация?"}],
  "user_id": "123",
  "ticket_id": "456"
}
```

## Принципы SOLID
- Single Responsibility: каждый модуль отвечает за свою область
- Open/Closed: расширяемость через новые типы CRM
- Liskov Substitution: интерфейсы для разных типов CRM
- Interface Segregation: отдельные интерфейсы для пользователей, тикетов, FAQ
- Dependency Inversion: зависимости через абстракции
